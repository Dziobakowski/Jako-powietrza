{\bfseries{doctest}} is designed to "{}just work"{} as much as possible. It also allows configuring how it is built with a set of identifiers.

The identifiers should be defined before the inclusion of the framework header.

Defining something {\ttfamily globally} means for every source file of the binary (executable / shared object).

That way the test runner doesn\textquotesingle{}t have to be implemented in the binary (executable / shared object) and can be reused from another binary where it is built and exported.

To export the test runner from a binary simply use \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}\`{}\`{}\`{}\+DOCTEST\+\_\+\+CONFIG\+\_\+\+IMPLEMENTATION\+\_\+\+IN\+\_\+\+DLL\`{}\`{}\`{}\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} together with \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}\`{}\`{}\`{}\+DOCTEST\+\_\+\+CONFIG\+\_\+\+IMPLEMENT\`{}\`{}\`{}\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} (or \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}\`{}\`{}\`{}\+DOCTEST\+\_\+\+CONFIG\+\_\+\+IMPLEMENT\+\_\+\+WITH\+\_\+\+MAIN\`{}\`{}\`{}\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} but then the other binaries will have to link to the executable) in whatever source file the test runner gets implemented into. Note that this identifier should not be defined in the other source files of the binary which exports the doctest test runner -\/ or there will be linker conflicts -\/ having the same symbols as both imported and exported within the same binary.

Checkout the \href{../../examples/executable_dll_and_plugin/}{\texttt{ {\bfseries{example}}}} -\/ it shows how to have the test runner implemented in a dll (and there are even tests in a plugin which is dynamically loaded).

This should be defined globally in binaries that import the symbols.

While {\bfseries{doctest}} fully supports the traditional, x\+Unit, style of class-\/based fixtures containing test case methods this is not the preferred style. Instead {\bfseries{doctest}} provides a powerful mechanism for nesting subcases within a test case. For a more detailed discussion and examples see the \href{tutorial.md\#test-cases-and-subcases}{\texttt{ {\bfseries{tutorial}}}}.

Test cases and subcases are very easy to use in practice\+:


\begin{DoxyItemize}
\item {\bfseries{TEST\+\_\+\+CASE(\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} {\itshape test name} \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*})}}
\item {\bfseries{SUBCASE(\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} {\itshape subcase name} \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*})}}
\end{DoxyItemize}

{\itshape test name} and {\itshape subcase name} are free form, quoted, strings. Test names don\textquotesingle{}t have to be unique within the {\bfseries{doctest}} executable. They should also be string literals.

It is possible to write test cases inside of class bodies in C++17 with the help of {\ttfamily \doxylink{doctest_2doctest_8h_afbb012c604e265d1e1e7b5279180d691}{TEST\+\_\+\+CASE\+\_\+\+CLASS()}} -\/ used just like {\ttfamily \doxylink{doctest_2doctest_8h_a8bc255ff55721571e878568d2b5dcb35}{TEST\+\_\+\+CASE()}} -\/ making testing private parts of classes easier.

Keep in mind that even though {\bfseries{doctest}} is \href{faq.md\#is-doctest-thread-aware}{\texttt{ {\bfseries{thread-\/safe}}}} -\/ using subcases has to be done only in the main test runner thread.

Test cases can also be parameterized -\/ see the \doxysectlink{md_doctest_2doc_2markdown_2parameterized-tests}{{\bfseries{documentation}}}{0}

Test cases and subcases can be filtered through the use of the \doxysectlink{md_doctest_2doc_2markdown_2commandline}{{\bfseries{command line}}}{0}\hypertarget{md_doctest_2doc_2markdown_2testcases_autotoc_md171}{}\doxysection{\texorpdfstring{BDD-\/style test cases}{BDD-\/style test cases}}\label{md_doctest_2doc_2markdown_2testcases_autotoc_md171}
In addition to {\bfseries{doctest}}\textquotesingle{}s take on the classic style of test cases, {\bfseries{doctest}} supports an alternative syntax that allow tests to be written as "{}executable specifications"{} (one of the early goals of \href{http://dannorth.net/introducing-bdd/}{\texttt{ Behaviour Driven Development}}). This set of macros map on to {\ttfamily TEST\+\_\+\+CASE}s and {\ttfamily SUBCASE}s, with a little internal support to make them smoother to work with.


\begin{DoxyItemize}
\item {\bfseries{SCENARIO(\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} {\itshape scenario name} \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*})}}
\end{DoxyItemize}

This macro maps onto {\ttfamily TEST\+\_\+\+CASE} and works in the same way, except that the test case name will be prefixed by "{}\+Scenario\+: "{}


\begin{DoxyItemize}
\item {\bfseries{SCENARIO\+\_\+\+TEMPLATE(\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} {\itshape scenario name}, {\itshape type}, {\itshape list of types} \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*})}}
\end{DoxyItemize}

This macro maps onto {\ttfamily TEST\+\_\+\+CASE\+\_\+\+TEMPLATE} and works in the same way, except that the test case name will be prefixed by "{}\+Scenario\+: "{}


\begin{DoxyItemize}
\item {\bfseries{SCENARIO\+\_\+\+TEMPLATE\+\_\+\+DEFINE(\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} {\itshape scenario name}, {\itshape type}, {\itshape id} \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*})}}
\end{DoxyItemize}

This macro maps onto {\ttfamily TEST\+\_\+\+CASE\+\_\+\+TEMPLATE\+\_\+\+DEFINE} and works in the same way, except that the test case name will be prefixed by "{}\+Scenario\+: "{}


\begin{DoxyItemize}
\item {\bfseries{GIVEN(\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} {\itshape something} \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*})}}
\item {\bfseries{WHEN(\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} {\itshape something} \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*})}}
\item {\bfseries{THEN(\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} {\itshape something} \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*})}}
\end{DoxyItemize}

These macros map onto {\ttfamily SUBCASE}s except that the subcase names are the \+\_\+something\+\_\+s prefixed by "{}given\+: "{}, "{}when\+: "{} or "{}then\+: "{} respectively.


\begin{DoxyItemize}
\item {\bfseries{AND\+\_\+\+WHEN(\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} {\itshape something} \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*})}}
\item {\bfseries{AND\+\_\+\+THEN(\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*} {\itshape something} \texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*})}}
\end{DoxyItemize}

Similar to {\ttfamily WHEN} and {\ttfamily THEN} except that the prefixes start with "{}and "{}. These are used to chain {\ttfamily WHEN}s and {\ttfamily THEN}s together.

When any of these macros are used the console reporter recognises them and formats the test case header such that the Givens, Whens and Thens are aligned to aid readability.

Other than the additional prefixes and the formatting in the console reporter these macros behave exactly as {\ttfamily TEST\+\_\+\+CASE}s and {\ttfamily SUBCASE}s. As such there is nothing enforcing the correct sequencing of these macros -\/ that\textquotesingle{}s up to the programmer!

Note that when using the \href{https://github.com/doctest/doctest/blob/master/doc/markdown/commandline.md}{\texttt{ {\ttfamily -\/-\/test-\/case=\texorpdfstring{$<$}{<}filters\texorpdfstring{$>$}{>}}}} command line option (or {\ttfamily -\/-\/subcase=\texorpdfstring{$<$}{<}filters\texorpdfstring{$>$}{>}}) you will have to pass the prefix {\ttfamily Scenario\+:} as well.\hypertarget{md_doctest_2doc_2markdown_2testcases_autotoc_md172}{}\doxysection{\texorpdfstring{Test fixtures}{Test fixtures}}\label{md_doctest_2doc_2markdown_2testcases_autotoc_md172}

Multiple decorators can be used at the same time. These are the currently supported decorators\+:


\begin{DoxyItemize}
\item {\bfseries{{\ttfamily skip(bool = true)}}} -\/ marks the test case to be skipped from execution -\/ unless the {\ttfamily -\/-\/no-\/skip} option is used
\item {\bfseries{{\ttfamily no\+\_\+breaks(bool = true)}}} -\/ no breaking into the debugger for asserts in the test case -\/ useful in combination with {\ttfamily may\+\_\+fail}/{\ttfamily should\+\_\+fail}/{\ttfamily expected\+\_\+failures}
\item {\bfseries{{\ttfamily no\+\_\+output(bool = true)}}} -\/ no output from asserts in the test case -\/ useful in combination with {\ttfamily may\+\_\+fail}/{\ttfamily should\+\_\+fail}/{\ttfamily expected\+\_\+failures}
\item {\bfseries{{\ttfamily may\+\_\+fail(bool = true)}}} -\/ doesn\textquotesingle{}t fail the test if any given assertion fails (but still reports it) -\/ this can be useful to flag a work-\/in-\/progress, or a known issue that you don\textquotesingle{}t want to immediately fix but still want to track in your tests
\item {\bfseries{{\ttfamily should\+\_\+fail(bool = true)}}} -\/ like {\bfseries{{\ttfamily may\+\_\+fail()}}} but fails the test if it passes -\/ this can be useful if you want to be notified of accidental, or third-\/party, fixes
\item {\bfseries{{\ttfamily expected\+\_\+failures(int)}}} -\/ defines the number of assertions that are expected to fail within the test case -\/ reported as failure when the number of failed assertions is different than the declared expected number of failures
\item {\bfseries{{\ttfamily timeout(double)}}} -\/ fails the test case if its execution exceeds this limit (in seconds) -\/ but doesn\textquotesingle{}t terminate it -\/ that would require subprocess support
\item {\bfseries{{\ttfamily test\+\_\+suite("{}name"{})}}} -\/ can be used on test cases to override (or just set) the test suite they are in
\item {\bfseries{{\ttfamily description("{}text"{})}}} -\/ a description of the test case
\end{DoxyItemize}

The values that the decorators take are computed while registering the test cases (during global initialization) -\/ before entering {\ttfamily \doxylink{how__exception__translators__work_8cpp_ae66f6b31b5ad750f1fe042a706a4e3d4}{main()}} and not just before running them.
